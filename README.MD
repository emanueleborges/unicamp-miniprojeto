# рҹ“ұ SensorMonitor

Aplicativo Android para monitoramento de sensores em tempo real, desenvolvido como **Mini Projeto** da disciplina de Desenvolvimento Mobile.

---

## рҹ“Ӣ Sobre o Projeto

O **SensorMonitor** utiliza sensores reais do dispositivo Android (acelerГҙmetro e sensor de luz) para capturar, exibir e armazenar dados sensoriais. O app segue a arquitetura **MVVM + Clean Architecture** com **Jetpack Compose** para UI, **Hilt** para injeГ§ГЈo de dependГӘncia e persistГӘncia com **Room** (Jetpack ORM sobre SQLite).

## рҹ–ҘпёҸ Telas do Aplicativo

### 1. Tela Principal (`MainScreen`)
- Menu com 3 cards clicГЎveis para navegar entre funcionalidades
- Cards coloridos para AcelerГҙmetro, Sensor de Luz e HistГіrico
- Hero card com status do sistema e badges

### 2. AcelerГҙmetro (`AccelerometerScreen`)
- ExibiГ§ГЈo em tempo real dos eixos **X**, **Y** e **Z** (m/sВІ)
- Barras de progresso coloridas para cada eixo
- CГЎlculo da **magnitude** do vetor aceleraГ§ГЈo
- **DetecГ§ГЈo de agitaГ§ГЈo (shake)** вҖ” vibra o dispositivo ao detectar movimento intenso
- BotГЈo para **salvar leitura** no banco de dados
- Estado gerenciado via `AccelerometerViewModel` + `StateFlow`

### 3. Sensor de Luz (`LightSensorScreen`)
- ExibiГ§ГЈo do valor de luminosidade em **lux**
- **Cor de fundo dinГўmica** вҖ” muda conforme o nГӯvel de luz ambiente:
  - рҹҢҷ Muito Escuro вҶ’ рҹҢ‘ Escuro вҶ’ рҹҸ  Interno вҶ’ вҳҒ Moderado вҶ’ рҹҢӨ Claro вҶ’ вҳҖ Muito Claro вҶ’ рҹ”Ҷ Luz Solar
- Barra de progresso proporcional
- BotГЈo para **salvar leitura** no banco de dados
- Estado gerenciado via `LightSensorViewModel` + `StateFlow`

### 4. HistГіrico (`HistoryScreen`)
- Lista todas as leituras salvas usando `LazyColumn`
- Indicador colorido por tipo de sensor
- Exibe valores e data/hora de cada registro
- BotГЈo de **lixeira** em cada item para **deletar individualmente**
- BotГЈo para **limpar todo o histГіrico**
- Estado gerenciado via `HistoryViewModel` + `StateFlow`

### рҹ§… Diagrama de Camadas вҖ” Clean Architecture

<img src="clean-arquiteture.png">

O projeto segue fielmente o **Diagrama de Camadas da Arquitetura Limpa (Clean Architecture)** de Robert C. Martin:

| Camada (Diagrama) | DescriГ§ГЈo | Mapeamento no Projeto |
|------------------------------------------|-----------------------------------------|----------------------------------------------------------------------------------|
| **Entidades** (centro вҖ” amarelo)         | Regras de NegГіcio da Empresa            | `data/model/SensorReading.kt` (`@Entity` Room)                                   |
| **Casos de Uso** (rosa)                  | Regras de NegГіcio da AplicaГ§ГЈo          | `domain/usecase/` вҖ” `SaveAccelerometerReadingUseCase`, `ClearAllReadingsUseCase` |

| **Adaptadores de Interface** (verde)     | Controladores, Apresentadores, Gateways | **Gateway:** `SensorRepositoryInterface` В·                                       |  
|                                                                                    | **Controladores/Apresentadores:** ViewModels (`AccelerometerViewModel`,          |
|                                                                                    | `LightSensorViewModel`, `HistoryViewModel`)                                      | 
|                                                                                    | **ImplementaГ§ГЈo:** `SensorRepository`                                            |
| **Frameworks & Drivers** (azul вҖ” externo)| DB, UI, Web, Dispositivos               | **DB:** `SensorDatabase` + `SensorDao` (Room)                                    | 
|                                                                                    | **UI:** Jetpack Compose Screens + `MainActivity`                                 | 
|                                                                                    | **Dispositivos:** `SensorManager`, `AccelerometerHandler`, `LightSensorHandler`  |

#### Regra de DependГӘncia

As dependГӘncias sГі apontam **para dentro** (da camada externa para a interna):

```
Frameworks & Drivers (azul)
  в””в”Җв”Җ Compose Screens / Room / SensorManager
        вҶ“
Adaptadores de Interface (verde)
  в””в”Җв”Җ ViewModels вҶ’ UseCases (nunca acessam o banco diretamente)
  в””в”Җв”Җ SensorRepository implementa SensorRepositoryInterface
        вҶ“
Casos de Uso (rosa)
  в””в”Җв”Җ Dependem apenas de SensorRepositoryInterface (contrato)
        вҶ“
Entidades (amarelo)
  в””в”Җв”Җ SensorReading вҖ” nГЈo depende de nenhuma outra camada
```

A inversГЈo de dependГӘncia via `SensorRepositoryInterface` no pacote `domain/` Г© exatamente o **Gateway** do diagrama, garantindo que a camada de domГӯnio nunca dependa de frameworks externos (Room, Android SDK).
